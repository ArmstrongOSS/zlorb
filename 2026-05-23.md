
## STRUCT: ServiceCoordinator
*The primary orchestrator. It manages the system's global state and the continuous operation loop.*
### PROPERTIES
*   **`service_config`**: `ServiceConfig` - The main operational settings (sleep time, credentials).
*   **`repo_configs`**: `Vec<Config>` - A collection of loaded configurations for all managed repositories.
*   **`first_run_flag`**: `bool` - Internal state tracking if the system has run previously in the cycle.
### METHODS
*   **`new(service_config: ServiceConfig)`**: Initializes the coordinator with the global service settings.
*   **`run_loop()`**:
    *   *Flow:* The main execution loop.
    *   *Steps:* 1. Calls `wait_for_run()`. 2. Calls `run_cycle()`. 3. Repeats indefinitely.
*   **`wait_for_run(first_run: bool)`**:
    *   *Logic:* Implements the necessary delay/sleep mechanism, handling the logic for the first execution cycle.
*   **`run_cycle()`**:
    *   *Logic:* Iterates through all `repo_configs`. For each config, it creates a `RepoProcessor` and executes the update cycle.
    *   *Returns:* `Result<(), ZlorbError>` indicating overall success or failure of the cycle.

## STRUCT: ConfigManager
*A utility service responsible solely for the persistence and loading of configuration files.*
### PROPERTIES
*   **`home_dir`**: `PathBuf` - The absolute path to the global application configuration directory.
### METHODS
*   **`initialize_default_config()`**:
    *   *Action:* Checks if the service config file exists. If not, creates a default `ServiceConfig` and writes it to disk.
*   **`load_service_config()`**:
    *   *Action:* Reads and deserializes the global `ServiceConfig` from `home_dir`.
*   **`load_repo_config(dir_entry: DirEntry)`**:
    *   *Action:* Constructs the full path to the `config.json` inside a given repository directory and deserializes it into a `Config` struct.

## STRUCT: RepoProcessor
*The core business logic unit. It encapsulates all actions related to a single Git repository's state synchronization.*
### PROPERTIES
*   **`repo_path`**: `PathBuf` - The local directory path of the repository to manage.
*   **`repo`**: `Repository` - The opened `git2::Repository` instance for the local clone.
*   **`config`**: `Config` - The configuration associated with this specific repository (branch, build command, etc.).
### METHODS
*   **`new(config: Config)`**: Initializes the `RepoProcessor` by opening the `git2::Repository` for the given `config.path`.
*   **`check_for_updates()`**:
    *   *Flow:* Coordinates the entire update sequence for the repo.
    *   *Steps:* 1. Calls `fetch_remote_updates()`. 2. Checks if the local branch is behind the remote. 3. If necessary, calls `run_build()`.
*   **`fetch_remote_updates()`**:
    *   *Logic:* Handles the full `git fetch` and fast-forward analysis required to bring the local branch up to date. (Handles `git2` error mapping).
*   **`run_build()`**:
    *   *Logic:* Calls the `BuildSystemExecutor` to execute the configured build command.

## STRUCT: BuildSystemExecutor
*An isolated utility service responsible for running external shell commands (the build step).*
### PROPERTIES
*   **`stdout_pipe`**: `Stdio` - Standard output stream for capturing build logs.
### METHODS
*   **`execute(target_path: &Path, command: &str)`**:
    *   *Action:* Sets the current working directory to `target_path`. Executes the `command` using `std::process::Command`.
    *   *Error Handling:* Checks the exit status code. Returns an error if the process fails (e.g., non-zero exit code).

## STRUCT: DirectoryScanner
*Utility service responsible for finding all relevant repository directories.*
### PROPERTIES
*   **`base_dir`**: `PathBuf` - The parent directory containing all sub-repositories.
### METHODS
*   **`scan_directories()`**:
    *   *Action:* Reads the contents of `base_dir`.
    *   *Returns:* `Result<Vec<DirEntry>, ZlorbError>` containing all directories that are potential repositories.
